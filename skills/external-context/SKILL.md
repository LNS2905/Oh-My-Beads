<Purpose>
Fetch external SDK, API, or library documentation on demand. Instead of a single
monolithic search, the skill decomposes the topic into 2–5 focused search facets,
dispatches parallel document-specialist subagents (each with WebSearch + WebFetch),
and synthesizes their findings into a single structured reference document with
citations. The output is saved to `.oh-my-beads/context/{topic}.md` so future
agents and sessions can reference it.
</Purpose>

<Use_When>
- User says "fetch docs for X", "find docs for X", "external context for X"
- User invokes `/oh-my-beads:external-context`
- An agent needs up-to-date SDK or API documentation not available in the codebase
- Research is needed before planning or implementing an unfamiliar integration
</Use_When>

<Do_Not_Use_When>
- Documentation is already available in the project (check first)
- The topic is a general programming concept (no external docs needed)
- The user just wants a code explanation, not external reference material
</Do_Not_Use_When>

<HARD-GATE>
**NEVER fabricate documentation.** Every fact in the output MUST come from a real
web source with a citation URL. If a specialist returns no useful results for a
facet, state "No authoritative source found" — do not invent content.
</HARD-GATE>

<HARD-GATE>
**ALWAYS save output to `.oh-my-beads/context/{topic}.md`.** The whole point of this
skill is persistent context. Never skip the write step, even for short results.
</HARD-GATE>

<Steps>

## Step 1: Parse the Topic

Extract the documentation topic from the user's request.

- Identify the **technology/library/API** name (e.g., "Stripe API", "React Router v7", "OpenAI SDK")
- Determine the **scope**: full overview, specific feature, migration guide, etc.
- Generate a **slug** for the output filename: lowercase, hyphens, no special chars
  (e.g., "stripe-api", "react-router-v7", "openai-sdk")

If the topic is ambiguous, ask ONE clarifying question before proceeding.

## Step 2: Decompose into Search Facets

Break the topic into **2–5 focused search facets**. Each facet targets a different
angle of the documentation need:

| Facet Type | Example |
|-----------|---------|
| **Getting Started** | "X getting started guide official docs" |
| **API Reference** | "X API reference methods parameters" |
| **Configuration** | "X configuration options setup" |
| **Examples** | "X code examples tutorials" |
| **Migration/Changelog** | "X migration guide v6 to v7 breaking changes" |

Guidelines for facet decomposition:
- Minimum 2 facets (overview + API reference)
- Maximum 5 facets (avoid diminishing returns)
- Each facet should be specific enough to yield targeted results
- Prefer official documentation domains when possible

## Step 3: Spawn Parallel Document Specialists

For each facet, spawn a subagent using the **Task** or **Agent** tool:

```
Task/Agent: Document Specialist — {facet_name}

You are a documentation researcher. Your task:

1. **Search** for: "{search_query}"
   Use WebSearch to find the most authoritative sources (prefer official docs).

2. **Fetch** the top 1–2 most relevant pages using WebFetch/FetchUrl.

3. **Extract** key information:
   - Core concepts and definitions
   - Function/method signatures with parameters
   - Configuration options and defaults
   - Code examples (keep them concise)
   - Version-specific notes

4. **Return** a structured summary in this format:

## {Facet Name}

**Sources:**
- [Title](URL)

### Key Information

{Extracted content — concise, factual, well-organized}

### Code Examples (if applicable)

{Brief, relevant code snippets from the docs}

IMPORTANT: Only include information you found in real web sources.
Do NOT fabricate or guess. Cite every claim.
```

**Spawn all specialists in parallel** (use multiple Task/Agent calls in the same
message) to minimize latency.

## Step 4: Synthesize Results

Once all specialists return:

1. **Collect** all specialist outputs
2. **Deduplicate** overlapping information across facets
3. **Organize** into a cohesive document structure:

```markdown
# {Topic} — External Context

> Auto-generated by oh-my-beads:external-context on {date}
> Query: "{original user query}"

## Overview
{High-level summary synthesized from all facets}

## {Section per major area}
{Merged and deduplicated content from specialists}

## Quick Reference
{Key APIs, methods, or config options in a compact table}

## Code Examples
{Best examples collected from specialists}

## Sources
- [Source 1 Title](URL)
- [Source 2 Title](URL)
- ...
```

4. **Quality check**: Remove any content that lacks a source citation.
   If a specialist returned nothing useful, note the gap.

## Step 5: Save to Context Directory

Write the synthesized document:

```
mkdir -p .oh-my-beads/context/
Write: .oh-my-beads/context/{slug}.md
```

Report to the user:
- Number of facets searched
- Number of sources cited
- File path where the document was saved
- Brief summary of what was found

</Steps>

<Output_Format>
After completion, report:

```
✅ External context saved to .oh-my-beads/context/{slug}.md

  Topic: {topic}
  Facets searched: {N}
  Sources cited: {M}
  
  Summary: {1-2 sentence overview of what was found}
```
</Output_Format>

<Red_Flags>
- **Fabricating content**: Never invent API signatures, config options, or behavior
- **Single-source dependency**: Try to cross-reference across at least 2 sources
- **Stale documentation**: Prefer recent/versioned docs; note version in output
- **Scope creep**: Stick to the requested topic; don't expand into tangential areas
- **Skipping the save**: Always write to `.oh-my-beads/context/` — that's the deliverable
</Red_Flags>
